\appsection{RequiredEvidence Schemas}
\label{app:required-evidence-schemas}

This appendix lists the \textit{RequiredEvidence} schemas referenced in Section \ref{sec:required-evidence-schemas} in full. Each schema is an authored, versioned artefact. The version field is mandatory, the schema hash is computed over the canonical serialisation of the schema bytes, and the evaluation harness records both fields per task. The on-chain NFTAA contract recognises a fixed set of schema versions, so a schema that is not registered cannot be used to authorise a transaction even if a valid validator signature accompanies it.

The three schemas below cover the three evaluation domains of the thesis: an NFT or access asset, a productive DeFi asset, and a treasury-governance proposal. We also include a deliberately weakened variant per domain that is used as the sensitivity condition described in Section \ref{sec:required-evidence-schemas}.

\subsection{NFT or access asset}
\label{app:schema-nft}

\begin{verbatim}
{
  "schema_id":      "icvp.required_evidence.nft_access",
  "schema_version": "1.0.0",
  "asset_class":    "erc721_nft",
  "intent":         "community_access | long_term_holding | collateralisation",
  "sources": [
    "Lohmer et al. 2024 (cite: Lohmer2024)",
    "Ante 2021 (cite: Ante2021)",
    "Dowling 2022 (cite: Dowling2022)",
    "Nadini et al. 2021 (cite: Nadini2021)",
    "Kong and Lin 2021 (cite: Kong2021)"
  ],
  "required_categories": {
    "collection_identity": {
      "ttl":        "30d",
      "provenance": "contract_address + chain_id + verified_source",
      "hard":       ["contract_verified == true"]
    },
    "metadata_integrity": {
      "ttl":        "30d",
      "provenance": "token_uri + storage_backend + content_hash",
      "hard":       ["content_hash_matches_on_chain == true",
                     "storage_backend in {ipfs, arweave, on_chain}"]
    },
    "rarity_and_traits": {
      "ttl":        "30d",
      "provenance": "indexer_name + indexer_version + snapshot_block",
      "hard":       ["trait_distribution_present == true"]
    },
    "market_depth_and_liquidity": {
      "ttl":        "5m",
      "provenance": "marketplace + api_endpoint + retrieval_timestamp",
      "hard":       ["listings_count >= 0",
                     "thirty_day_volume_usd >= 0"]
    },
    "ownership_history": {
      "ttl":        "30d",
      "provenance": "indexer_name + start_block + end_block",
      "hard":       ["unique_holders >= 1"]
    }
  },
  "optional_categories": {
    "community_signals":    { "ttl": "7d",  "provenance": "platform + url" },
    "creator_track_record": { "ttl": "30d", "provenance": "source + url" }
  },
  "hard_constraints": [
    "intent in {community_access, long_term_holding, collateralisation}",
    "asset_class == erc721_nft"
  ]
}
\end{verbatim}

A weakened variant \texttt{icvp.required\_evidence.nft\_access@0.9.0-weak} demotes \texttt{rarity\_and\_traits} and \texttt{ownership\_history} to optional, removes the \texttt{content\_hash\_matches\_on\_chain} hard constraint, and loosens \texttt{market\_depth\_and\_liquidity} TTL from five minutes to one hour. This variant is used as the sensitivity condition.

\subsection{Versioning and reproducibility}

Each schema is checked into the prototype repository under \texttt{schemas/} and tagged with its \texttt{schema\_version}. The schema hash is computed over the canonical JSON serialisation. The evaluation harness loads schemas by version from a configuration field and records the loaded version and its hash in every task result, so that any reported number is reproducible against the same schema bytes. The on-chain NFTAA contract carries a registry of recognised schema versions and their hashes, so that a schema that is not registered cannot anchor an authorisation even if a valid validator signature accompanies it. This makes schema authorship a defended part of the contribution rather than an implementation detail.
